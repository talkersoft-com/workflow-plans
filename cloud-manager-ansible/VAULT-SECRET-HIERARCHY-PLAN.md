# VAULT-SECRET-HIERARCHY-PLAN — Consistent VM-centric secret paths

## Problem

Vault secrets for a VM are currently spread across two incompatible path schemes:

```
cloudmanager/vm/instances/vm_31K8J7WQFP          ← SSH key (flat, field: ssh_key)
cloudmanager/playbook-secrets/run_FSK3WSX404/pg-test/postgres  ← postgres (run-scoped)
cloudmanager/playbook-secrets/pg-test/postgres    ← postgres (stable, by VM name)
```

Three problems with the current layout:

1. **Two hierarchies for the same concept** — SSH lives under `vm/instances/` (keyed by public_id), postgres lives under `playbook-secrets/` (keyed by run_id or VM name). There is no single place to look up "all secrets for a VM".
2. **Run_id pollutes the canonical path** — `playbook-secrets/<run_id>/<vm_name>/postgres` exists purely as a token scoping artifact (the per-run Vault token can only write under its own prefix). It is not a meaningful storage location; it leaks internal plumbing into the secret namespace.
3. **VM name instead of public_id** — the stable path `playbook-secrets/<vm_name>/postgres` breaks if the VM is renamed, and cannot be located without knowing the name. The SSH key correctly uses the immutable `vm_public_id`.

## Goal

Every VM-scoped secret lives under a single hierarchy keyed by `vm_public_id`:

```
cloudmanager/vm/instances/<vm_public_id>/ssh
cloudmanager/vm/instances/<vm_public_id>/postgres
cloudmanager/vm/instances/<vm_public_id>/redis
cloudmanager/vm/instances/<vm_public_id>/nginx
```

`vault kv list cloudmanager/vm/instances/vm_31K8J7WQFP/` returns every secret the VM has, regardless of which playbook wrote it. The VM public_id is always the stable key — not the name, not the run_id.

## New path conventions

| Secret | Path | Written by |
|--------|------|------------|
| SSH keypair | `cloudmanager/vm/instances/<vm_public_id>/ssh` | vorch-lib at provision |
| PostgreSQL creds | `cloudmanager/vm/instances/<vm_public_id>/postgres` | pg playbook |
| Redis creds | `cloudmanager/vm/instances/<vm_public_id>/redis` | redis playbook (future) |
| Any playbook output | `cloudmanager/vm/instances/<vm_public_id>/<secret_type>` | playbook |

The per-run path (`playbook-secrets/<run_id>/...`) is **eliminated**. The stable path (`playbook-secrets/<vm_name>/...`) is **eliminated**. The `playbook-secrets/` subtree goes away entirely.

## Changes required

### Phase 1 — vorch-lib: move SSH key path

**File**: `vorch-lib/vault/client.go`

`StoreSSHKey`, `GetSSHKey`, and `DeleteSSHKey` currently build the path as `{basePath}/{vmName}`. Change to `{basePath}/{vmName}/ssh`.

```go
// Before
path := fmt.Sprintf("%s/%s", basePath, vmName)

// After
path := fmt.Sprintf("%s/%s/ssh", basePath, vmName)
```

The stored fields (`private_key`, `public_key`, `created_at` inside the `ssh_key` JSON blob) are unchanged. Only the KV path moves one level deeper.

Also update `GetSSHKey` and `DeleteSSHKey` in the same way — both use the same `path` construction.

The caller (`vorch-service provision.go`) passes `basePath = "cloudmanager/vm/instances"` and `vmName = vm.PublicId`. No change needed at the call site — the path construction is inside the library.

### Phase 2 — vorch-service: update SSH key read path

**File**: `vorch-service/internal/ansible/playbook_run_exec.go`

The `executeOneTarget` function reads the SSH key at `cloudmanager/vm/instances/<vmId>`. After Phase 1, the path is `cloudmanager/vm/instances/<vmId>/ssh`.

```go
// Before
sshSecret, err := h.Vault.ReadKVv2(ctx, "cloudmanager", fmt.Sprintf("vm/instances/%s", target.VMId))

// After
sshSecret, err := h.Vault.ReadKVv2(ctx, "cloudmanager", fmt.Sprintf("vm/instances/%s/ssh", target.VMId))
```

The field extraction logic after the read is unchanged — still reads `ssh_key` blob → unmarshal → `private_key`.

### Phase 3 — Vault policy: update worker token scope

**Target**: `cloudmanager-playbook-runner` policy (the worker's own long-lived token).

Add read access to the full `vm/instances/+/*` subtree (so the worker can read SSH keys and any other per-VM secrets it needs to resolve):

```hcl
# existing — keep
path "auth/token/create" { capabilities = ["update"] }
path "auth/token/create/playbook-run" { capabilities = ["update"] }
path "auth/token/revoke" { capabilities = ["update"] }
path "auth/token/revoke-accessor" { capabilities = ["update"] }

# existing read-back of playbook outputs — REMOVE (playbook-secrets subtree gone)
# path "cloudmanager/metadata/playbook-secrets/+/*" { ... }
# path "cloudmanager/data/playbook-secrets/+/*" { ... }

# new — read any per-VM secret (SSH key, playbook outputs)
path "cloudmanager/data/vm/instances/+/*" {
  capabilities = ["read"]
}
path "cloudmanager/metadata/vm/instances/+/*" {
  capabilities = ["read", "list"]
}
```

**Target**: per-run policy generated by `BuildRunPolicy` in `PlaybookRunService.cs`.

Currently scoped to `playbook-secrets/<run_id>/*`. Change to `vm/instances/<vm_public_id>/*` for each target VM:

```hcl
# Before
path "cloudmanager/data/playbook-secrets/<run_id>/*" {
  capabilities = ["create", "update", "read"]
}

# After — one stanza per target VM
path "cloudmanager/data/vm/instances/<vm_public_id>/*" {
  capabilities = ["create", "update", "read"]
}
path "cloudmanager/metadata/vm/instances/<vm_public_id>/*" {
  capabilities = ["read", "list"]
}
```

The `playbook-run` base role policy and the `playbook-run` token role itself are unchanged — the per-run policy is still created fresh per run and attached to a short-lived token.

### Phase 4 — cloud-manager-api: update BuildRunPolicy

**File**: `cloud-manager-api/src/Services/CloudManager.Data.Services/PlaybookRunService.cs`

`BuildRunPolicy` currently generates a policy scoped to `playbook-secrets/<runId>/*` plus per-VM name stable paths. Replace with `vm/instances/<vm_public_id>/*`:

```csharp
private static string BuildRunPolicy(string runPublicId, IEnumerable<(string PublicId, string Name)> vms)
{
    var sb = new StringBuilder();
    foreach (var vm in vms.Where(v => !string.IsNullOrWhiteSpace(v.PublicId)).DistinctBy(v => v.PublicId))
    {
        sb.Append($@"
path ""cloudmanager/data/vm/instances/{vm.PublicId}/*"" {{
  capabilities = [""create"", ""update"", ""read""]
}}
path ""cloudmanager/metadata/vm/instances/{vm.PublicId}/*"" {{
  capabilities = [""read"", ""list""]
}}");
    }
    return sb.ToString().Trim();
}
```

Update `TriggerMultiVmAsync` to pass both `PublicId` and `Name`:

```csharp
var vmTuples = vms.Select(v => (v.PublicId ?? "", v.Name));
await _vaultClient.WritePolicyAsync(policyName, BuildRunPolicy(run.PublicId, vmTuples));
```

Note: `run.PublicId` is no longer used in the policy body (the run_id prefix goes away). The `policyName` is still `playbook-run-<run_public_id>` — that naming is fine to keep for auditability.

Also remove `VaultSecretsPrefix` from the run record or keep it pointing to the first VM's path for backwards compatibility with any callers that read it.

### Phase 5 — Playbooks: update vault write paths

**Files**: `pg-14-jammy` and `pg-16-noble` playbook content in the database (updated via `cloud_playbook_set` or directly through the API).

Replace the two-path stable+run scheme with a single VM-anchored path:

```yaml
vars:
  # Before
  vault_stable_path: "playbook-secrets/{{ inventory_hostname }}/postgres"
  vault_run_path:    "playbook-secrets/{{ run_public_id }}/{{ inventory_hostname }}/postgres"

  # After — single path, keyed by vm_public_id (injected as extravar by porch)
  vault_path: "vm/instances/{{ vm_public_id }}/postgres"
```

The `vault_kv2_get` (idempotent read on rerun) and both `vault_kv2_write` tasks collapse into one:

```yaml
- name: Check for existing password in Vault
  community.hashi_vault.vault_kv2_get:
    url: "{{ vault_addr }}"
    auth_method: token
    token: "{{ vault_token }}"
    engine_mount_point: cloudmanager
    path: "{{ vault_path }}"
  delegate_to: localhost
  become: false
  register: vault_existing
  ignore_errors: true

- name: Resolve password — reuse existing or generate new
  ansible.builtin.set_fact:
    pg_resolved_password: >-
      {{ vault_existing.secret.password
         if (not vault_existing.failed
             and vault_existing.secret is defined
             and vault_existing.secret.password is defined)
         else lookup('password', '/dev/null length=32 chars=ascii_letters,digits') }}

# ... install postgres, create user, grant ...

- name: Store credentials in Vault
  community.hashi_vault.vault_kv2_write:
    url: "{{ vault_addr }}"
    auth_method: token
    token: "{{ vault_token }}"
    engine_mount_point: cloudmanager
    path: "{{ vault_path }}"
    data:
      host: "{{ ansible_default_ipv4.address }}"
      port: 5432
      db: "{{ pg_db_name }}"
      user: "{{ pg_app_user }}"
      password: "{{ pg_resolved_password }}"
  delegate_to: localhost
  become: false

- name: Surface the Vault path on the run so cloud-manager can show it
  ansible.builtin.set_fact:
    password: "vault:cloudmanager/data/{{ vault_path }}"
```

One read, one write. No run_id in the path. Same password on every rerun.

### Phase 6 — Vault policy cleanup

Remove the now-unused `playbook-run` base policy (the broad wildcard on `playbook-secrets/*`) and the `cloudmanager-playbook-write` policy. Delete any leftover `playbook-run-run_*` per-run policies via the vault-toolkit MCP after migration.

## Migration

VMs have been deleted. There are no live secrets under `playbook-secrets/` or `vm/instances/` to migrate. The implementation can proceed against a clean Vault state.

After Phase 1 ships, new VM provisioning will write SSH keys to the new `/ssh` subpath. After Phase 5 ships, playbook runs will write postgres creds to `vm/instances/<vm_public_id>/postgres`.

## Execution order

Phases 1–2 (SSH path) and Phases 3–5 (playbook path) are independent and can be implemented in parallel, but both must land before VMs are provisioned again.

Phase 6 (cleanup) runs last, after confirming no secrets remain at the old paths.
